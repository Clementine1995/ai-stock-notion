# 股市舆情与公告分析工具 PRD

## 1. 背景

本项目面向 A 股短线交易研究，目标是构建一个低成本、可持续迭代的 AI 研究助理。系统通过采集公告、新闻、外围市场、行情数据以及用户沉淀在 Notion 中的交易经验，生成盘后复盘、盘前交易指导、午间复盘报告，帮助用户发现预期差板块、识别风险、沉淀可复用经验。

系统不做自动交易，不输出无条件买卖指令。所有结论必须带有证据、置信度、成立条件和失效条件，由用户最终决策。

## 2. 目标

### 2.1 产品目标

- 每个交易日自动生成三类报告：盘后复盘、盘前报告、午间复盘。
- 从公告、新闻、行情、外围市场和个人知识库中提取交易相关信息。
- 识别可能存在预期差的板块、个股和风险点。
- 建立预测与实际走势的验证闭环，让系统持续总结有效规则和失效规则。
- 支持逐步扩展数据源、分析模型和 Web UI。

### 2.2 MVP 目标

MVP 阶段只要求跑通完整闭环：

1. 同步用户 Notion 经验文档。
2. 采集 A 股公告、新闻、基础行情数据。
3. 生成盘后、盘前、午间三类 Markdown 报告。
4. 每条重点结论保留来源、证据、置信度和验证条件。
5. 记录观察项，并在次日或后续交易日进行结果回填。

### 2.3 非目标

MVP 阶段不做以下内容：

- 自动交易或自动下单。
- 秒级实时盯盘。
- 高频量化策略。
- 复杂可视化大屏。
- 多 Agent 协同系统。
- 对所有研报 PDF 的深度结构化解析。

## 3. 用户与场景

### 3.1 目标用户

主要用户是具备一定短线交易经验、关注题材轮动和价值投机的个人投资者。

用户已有部分交易经验文档，存放在 Notion，希望 AI 能结合这些经验进行复盘和推演，而不是只根据通用金融知识生成报告。

### 3.2 核心使用场景

#### 盘后复盘

时间：交易日 15:30-21:00。

目标：

- 总结当日市场情绪、强弱板块、涨跌停结构。
- 汇总盘后公告、重大新闻和异常事件。
- 对今日盘前推演进行验证。
- 形成次日关注方向。

#### 盘前报告

时间：交易日 07:30-08:50。

目标：

- 分析隔夜外围市场、政策、产业消息、公告、研报摘要。
- 识别可能存在预期差的板块和标的。
- 给出开盘验证条件和不成立条件。

#### 午间复盘

时间：交易日 11:40-12:30。

目标：

- 判断上午市场是否符合盘前推演。
- 识别下午可能加强、切换或退潮的方向。
- 提醒风险，不追求实时看盘精度。

#### 周度复盘

时间：周五收盘后或周末。

目标：

- 汇总本周预测命中情况。
- 找出有效规则、失效规则和误判原因。
- 更新经验库建议。

## 4. 核心需求

### 4.1 数据采集

系统需要采集以下数据：

| 数据类型 | MVP 优先级 | 示例内容 |
| --- | --- | --- |
| 上市公司公告 | P0 | 业绩预告、重大合同、减持、回购、并购、监管 |
| 新闻舆情 | P0 | 政策、产业、公司、海外映射 |
| 行情数据 | P0 | 指数、个股日线、成交额、涨跌幅、涨跌停 |
| Notion 经验文档 | P0 | 交易规则、历史案例、复盘笔记 |
| 外围市场 | P1 | 美股、港股、期货、汇率、商品 |
| 研报摘要 | P1 | 标题、摘要、行业观点 |
| 龙虎榜 | P2 | 游资、机构席位、净买入 |

### 4.2 知识库

知识库需要支持：

- 从 Notion 同步页面内容。
- 将文档按段落或语义块切分。
- 生成 embedding 并写入向量库。
- 保留来源页面、标题、更新时间、标签。
- 查询时返回相似经验和历史案例。

知识库分类：

- 交易原则库。
- 历史案例库。
- 每日复盘库。
- 错误与反思库。

### 4.3 信息结构化

系统需要将公告、新闻、研报等非结构化文本转换为结构化事件：

```json
{
  "event_type": "earnings_forecast",
  "impact_direction": "positive",
  "affected_sectors": ["算力", "半导体"],
  "related_stocks": ["000001.SZ"],
  "summary": "一句话摘要",
  "evidence": ["原文关键证据"],
  "expectation_gap_score": 4,
  "risk_score": 2,
  "confidence": 0.78
}
```

### 4.4 分析评分

每个观察方向至少包含以下评分：

| 指标 | 说明 |
| --- | --- |
| 催化强度 | 消息或公告本身是否足够强 |
| 新鲜度 | 市场是否尚未充分交易 |
| 预期差 | 是否可能超出市场已有认知 |
| 板块扩散潜力 | 是否容易带动产业链或同类公司 |
| 流动性 | 是否具备短线资金参与基础 |
| 风险强度 | 是否存在高位兑现、监管、证伪等风险 |
| 综合优先级 | A/B/C，不等同于买卖建议 |

### 4.5 报告输出

#### 盘后复盘报告

必须包含：

- 市场概况。
- 情绪指标。
- 强势板块。
- 弱势板块。
- 重要公告和新闻。
- 今日推演验证。
- 明日观察方向。
- 风险清单。

#### 盘前报告

必须包含：

- 隔夜外围市场。
- 隔夜重大消息。
- 可能存在预期差的板块。
- 观察标的。
- 开盘验证条件。
- 失效条件。
- 风险提示。

#### 午间复盘报告

必须包含：

- 上午市场与盘前推演对比。
- 主线是否加强、分歧或切换。
- 下午机会。
- 下午风险。
- 应放弃或降低关注的方向。

### 4.6 验证闭环

系统需要记录每次报告中的观察项：

```json
{
  "trade_date": "2026-05-27",
  "report_type": "pre_market",
  "theme": "算力",
  "related_stocks": ["000001.SZ"],
  "hypothesis": "隔夜公告可能带来预期差",
  "validation_condition": "板块开盘成交额放大且核心标的高开不回落",
  "invalid_condition": "核心标的低开低走或板块无跟随",
  "priority": "A",
  "status": "pending"
}
```

后续回填：

- 是否命中。
- 是否失效。
- 实际行情表现。
- 误判原因。
- 可复用经验。

## 5. 约束与原则

### 5.1 安全边界

- 不生成自动买入、卖出、仓位指令。
- 不承诺收益。
- 不将 AI 结论作为事实，必须标注证据和置信度。
- 结论必须包含不成立条件。

### 5.2 成本约束

- MVP 优先使用开源工具和本地部署。
- 数据库使用 PostgreSQL。
- 向量库使用本地 Qdrant。
- 报告先使用 Markdown，不先做复杂前端。

### 5.3 质量原则

- 事实与推断分离。
- 结论可追溯到来源。
- 每个报告可以被第二天验证。
- 每周进行规则复盘，不让模型无限自我强化错误经验。

## 6. 成功指标

### 6.1 MVP 验收指标

- 可以在本地一键运行采集、分析、报告生成流程。
- 每日三类报告均可生成。
- 80% 以上重点结论包含证据来源。
- 观察项可被记录并回填结果。
- Notion 至少一个经验库页面可成功同步并参与检索。

### 6.2 中期指标

- 连续运行 20 个交易日。
- 每周自动生成命中率和误判总结。
- 用户可以快速检索历史相似案例。
- 盘前报告中的 A 类观察项具备稳定复盘价值。
